content: ## Context

Currently, projects are created by cloning the default branch from a remote repository. The VCS layer in `src/vcs/index.ts` supports cloning but doesn't handle branch selection. The session creation process in `src/sessions/routes.tsx` handles repository setup but assumes default branch behavior.

The Project model stores repository information but lacks branch-related fields. The UI forms for project creation need to collect optional branch preferences from users.

## Goals / Non-Goals

**Goals:**
- Allow users to optionally specify a source branch to clone from (empty = repo default)
- Allow users to optionally create a new branch locally for the project
- Support both Git and Fossil VCS systems with their respective branch semantics
- Create branches during project initialization, not pushed immediately
- Branch info is immutable after project creation

**Non-Goals:**
- Branch validation against remote (user provides exact branch names)
- Immediate push of new branch (handled by existing Commit action)
- Editable branch settings after project creation
- Branch protection or permission checks

## Decisions

### 1. Branch Creation Timing
**Decision**: Create branch during session initialization, not during project creation.
**Rationale**: Project creation only stores metadata; actual repository clone happens when first session is created. This keeps project creation lightweight.

### 2. No Branch Validation
**Decision**: Do not validate branch existence before cloning.
**Rationale**: Let the underlying VCS commands fail naturally if branch doesn't exist. This matches current behavior with repo URLs.

### 3. Git vs Fossil Handling
**Decision**: Handle branch operations differently for each VCS:
- Git: Use `--branch` flag for clone, `checkout -b` for new branch
- Fossil: Clone full repo, then `checkout` for source branch, `branch new` for new branch
**Rationale**: Fossil has different branching model - all branches exist in the repo.

### 4. Force Behavior
**Decision**: For Git, if new branch exists locally, overwrite. For Fossil, branch creation updates if exists.
**Rationale**: User wants to use this branch; if it exists, they want to work on it.

## Risks / Trade-offs

- **Risk**: Invalid branch names cause clone failures
  - **Mitigation**: User sees VCS error message; can recreate project with correct name

- **Risk**: Branch name collisions if multiple projects use same newBranch name
  - **Mitigation**: Each project has isolated upstream directory; branches don't conflict

- **Risk**: Fossil `branch new` behavior differs from Git
  - **Mitigation**: Document behavior differences; Fossil updates existing branches gracefully

- **Trade-off**: No remote validation means user might typo branch name
  - **Acceptance**: Consistent with current URL behavior (we don't validate URLs exist)

<content>1: ## Context
2: 
3: Currently, projects are created by cloning the default branch from a remote repository. The VCS layer in `src/vcs/index.ts` supports cloning but doesn't handle branch selection. The session creation process in `src/sessions/routes.tsx` handles repository setup but assumes default branch behavior.
4: 
5: The Project model stores repository information but lacks branch-related fields. The UI forms for project creation need to collect optional branch preferences from users.
6: 
7: ## Goals / Non-Goals
8: 
9: **Goals:**
10: - Allow users to optionally specify a source branch to clone from (empty = repo default)
11: - Allow users to optionally create a new branch locally for the project
12: - Support both Git and Fossil VCS systems with their respective branch semantics
13: - Create branches during project initialization, not pushed immediately
14: - Branch info is immutable after project creation
15: 
16: **Non-Goals:**
17: - Branch validation against remote (user provides exact branch names)
18: - Immediate push of new branch (handled by existing Commit action)
19: - Editable branch settings after project creation
20: - Branch protection or permission checks
21: 
22: ## Decisions
23: 
24: ### 1. Branch Creation Timing
25: **Decision**: Create branch during session initialization, not during project creation.
26: **Rationale**: Project creation only stores metadata; actual repository clone happens when first session is created. This keeps project creation lightweight.
27: 
28: ### 2. No Branch Validation
29: **Decision**: Do not validate branch existence before cloning.
30: **Rationale**: Let the underlying VCS commands fail naturally if branch doesn't exist. This matches current behavior with repo URLs.
31: 
32: ### 3. Git vs Fossil Handling
33: **Decision**: Handle branch operations differently for each VCS:
34: - Git: Use `--branch` flag for clone, `checkout -b` for new branch
35: - Fossil: Clone full repo, then `checkout` for source branch, `branch new` for new branch
36: **Rationale**: Fossil has different branching model - all branches exist in the repo.
37: 
38: ### 4. Force Behavior
39: **Decision**: For Git, if new branch exists locally, overwrite. For Fossil, branch creation updates if exists.
40: **Rationale**: User wants to use this branch; if it exists, they want to work on it.
41: 
42: ## Risks / Trade-offs
43: 
44: - **Risk**: Invalid branch names cause clone failures
45:   - **Mitigation**: User sees VCS error message; can recreate project with correct name
46: 
47: - **Risk**: Branch name collisions if multiple projects use same newBranch name
48:   - **Mitigation**: Each project has isolated upstream directory; branches don't conflict
49: 
50: - **Risk**: Fossil `branch new` behavior differs from Git
51:   - **Mitigation**: Document behavior differences; Fossil updates existing branches gracefully
52: 
53: - **Trade-off**: No remote validation means user might typo branch name
54:   - **Acceptance**: Consistent with current URL behavior (we don't validate URLs exist)
